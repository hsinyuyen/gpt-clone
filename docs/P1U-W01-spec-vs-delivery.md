# P1U-W01：書面規格 vs 交付成品 — 差異分析與給規格團隊的建議

> 對象：**教案 / 互動設計規格團隊**，以及專案負責人。
> 比對來源：
> - **書面 spec**：《P1上 W01 互動設計規格｜新手登入：我的第一次開機》（最後修改 2026-07-01）
> - **交付成品**：`public/games/p1u-w01-mouse.html`《點點鼠與海底寶藏城：滑鼠大冒險》（本次開發，VERSION 6，已上線）
> - 姊妹文件：[lesson-design-guide.md](lesson-design-guide.md)（較高年級 Lab Terminal 概念課的設計指南）

---

## 0. 一句話結論

> **交付的成品與書面 spec 是「同一個課槽 P1U-W01 的兩套不同設計」，技能覆蓋幾乎不重疊（交集只有『單擊』）。**
> 這不是「改了幾個細節」，而是主題、技能分類、站別結構、教學法、老師端、時長全部不同。
> **第一步不是改程式，是先開一次對齊會議，決定哪一套才是 P1U-W01 的正典。**

這份文件不評價對錯，而是把差異攤開、找出為什麼會這樣、並給規格團隊「下次怎麼寫 spec 能避免這種無人察覺的偏離」。

## 1. 逐面向對照表

| 面向 | 書面 spec《新手登入》 | 交付成品《海底寶藏城》 | 差異 |
|---|---|---|---|
| **主題/情境** | 第一次開機、登入自己的電腦（寫實桌面） | 海底寶藏城、滑鼠魔法冒險（奇幻） | 完全不同 |
| **核心目標** | 用滑鼠**自己登入電腦** | 學會 6–7 種滑鼠**操作**＋打 Boss＋閱讀理解 | 完全不同 |
| **技能分類** | CLICK／MOVE(游標追蹤)／TRACE(路徑)／COMBO(登入組合) | 單擊／雙擊／拖拉／滾輪／右鍵／左右鍵辨識／看指令＋(Boss)攻擊/重擊/防禦/魔法/閃避＋(王城)問AI/方向 | **幾乎不重疊** |
| **站別結構** | 5 站（30＋32＋30＋26＋10 次反覆） | 7 技能站＋10 關主節奏 Boss＋5 關王城閱讀理解 | 大幅擴張 |
| **教學法** | 動作技能週：反覆累積次數、肌肉記憶、失誤靜音、判定放大、花朵進度 | 節奏遊戲(縮圈打點)、動詞戰鬥、劇情、閱讀理解選擇題 | 由「肌肉記憶反覆」→「節奏＋戰鬥＋認知」 |
| **AI 使用** | ❌ 本週明確**不使用** | ✅ 王城關卡核心即「問 AI」＋全程 AI 生圖 | 與 spec 相反 |
| **文字量** | 極端精簡、**≤5 字**、圖示為主 | 有劇情、指令卡、閱讀理解選擇題（需識字） | 與 spec 相反 |
| **時長** | 90 分（60 任務＋30 自由玩） | 未對齊；內容量遠超，恐超時 | 需重估 |
| **老師端** | 站3 目標控制器、全班進度、🟢🟡🔴 狀態、小老師派任 | ❌ 遊戲內無 teacher panel | 缺漏 |
| **反「硬 A」語義** | 防**假動作**（400px 位移、3s 沿線、1s 過場） | 防**挫折**（非致命、逾時放行、連錯放行） | 兩種不同語義 |
| **資料結構** | `s1_wakeup…s5_login` 明確 schema＋舉手/小老師欄位 | `s1–s8＋s8round＋cstep＋gameProgress`（本機＋帳號雲端） | 不同；成品有續玩存檔 |
| **金幣** | 250 | 310（250＋王城 60） | 超出 spec |
| **Deliver 時刻** | 撒花＋徽章＋250金幣＋「舉手告訴老師」＋「我是小老師」翻轉教育 | 撒花＋徽章＋金幣（**無**舉手/小老師/翻轉教育） | 部分缺漏 |
| **判定放大 (M4)** | 所有點擊判定 ≥100px | 有大判定區（機制不同） | 精神一致、實作不同 |

## 2. 最關鍵的發現：技能覆蓋幾乎不重疊

```
spec 要教：   CLICK  ·  MOVE(四角探險)  ·  TRACE(滑鼠迷宮)  ·  COMBO(登入→桌面→登出)
成品在教：    單擊  ·  雙擊  ·  右鍵  ·  滾輪  ·  拖拉  ·  左右鍵辨識  ·  看指令做動作
              ＋ Boss：攻擊/重擊/防禦/魔法/閃避     ＋ 王城：問AI / 上下左右+四對角
              ─────────────────────────────────
交集 ≈        只有「單擊 / CLICK」
```

- **spec 刻意把第一週收斂在 click／move／trace**（文件寫明「學期第 1 週，起點，無回顧、無承接」），因為這是小一**第一次用電腦**。
- **成品把疑似整個模組一（W01–W06）的滑鼠技能全塞進 W01**：雙擊、右鍵、滾輪、拖拉一次全上。
- spec 的三種核心站型（四角探險 MOVE、滑鼠迷宮 TRACE、登入驗收 COMBO）在成品中**完全不存在**。

> 這代表兩者不是「同一課的不同版本」，比較像「兩堂不同的課」。

---

## 3. 為什麼會走到這裡（中性歸因，非究責）

1. **成品先於對齊存在**：交付的奇幻滑鼠遊戲在本次對話前就已存在，之後被一連串即時聊天需求（「做成真正的遊戲」「擴充到 10 關」「加戰鬥」「Boss 後加閱讀理解」）推著長大——**每一步都沒有回到這份書面 spec 對齊**。
2. **spec 與實作沒有追溯關係**：文件末尾建議「本文件的副本放進 `p1u-w01/`」，但實作是單檔 `p1u-w01-mouse.html`，兩者沒有互相指向。偏離因此無人察覺。
3. **spec 幾乎與開發同時完成**（spec 標注 2026-07-01，成品同期開發）：很可能是**兩條平行的設計線**，從未交會。
4. **定性需求推動範圍蔓延**：「像專業遊戲」「情緒價值給滿」這類形容詞沒有可驗收邊界，工程端只能往「更多內容 / 更炫」的方向猜，於是越做越大、越偏越遠。

---

## 4. 交付品「多做了」與「少做/沒做」清單

**多做（超出 spec）**
- 6 種離散滑鼠操作（雙擊／右鍵／滾輪／拖拉／左右鍵／看指令）
- 10 位關主 × 5 動詞的節奏戰鬥 Boss（含 POV 動畫、暴怒二階段）
- 5 關「王城入關登記」閱讀理解＋問 AI＋方向認知（上下左右＋四對角）
- 15＋ 張 AI 生圖、本機＋帳號雲端**存檔續玩**、金幣入帳號錢包、後台多班級可見

**少做 / 沒做（spec 有、成品沒有）**
- 三種核心站型：**MOVE 四角探險、TRACE 滑鼠迷宮、COMBO 登入驗收**
- 「開機 → 登入」的核心情境（成品是奇幻探寶，非登入自己的電腦）
- Deliver 的**翻轉教育**：舉手告訴老師、「我是小老師」派任（＋125 金幣）
- **老師端**：站3 目標即時同步、全班進度總覽、需協助/舉手狀態
- spec 的 **M1 動作品質驗證**語義（400px 位移／3s 沿線防假動作）

## 5. 需要 spec 團隊實質留意的取捨與風險

1. **首週範圍過載**：6–7 歲第一次用電腦，一次教雙擊／右鍵／滾輪／拖拉可能過重。spec 的收斂（只 click/move/trace）在教學上其實更站得住腳。
2. **教學法安全性**：spec 的 **M2「失誤靜音、不用紅框否定、小一玻璃心」** 對這個年齡更保守。成品雖有非致命保底，但 **Boss 紅色扣血、暴怒紅光、縮圈時間壓力** 與「不否定、不製造壓力」的精神相左，需確認可接受度。
3. **識字/認知負荷**：spec 明訂「不使用 AI、文字 ≤5 字」；成品的**閱讀理解選擇題**假設學生能讀題。小一多數仍在識字初期，這是風險，至少要標注前置識字假設。
4. **時長**：spec 是 90 分；成品（7 站＋10 關主＋5 閱讀關）內容量遠超，很可能超時，需重新估時。
5. **課堂運作**：老師端與翻轉教育是 spec 設計的**課堂骨幹**；成品缺這塊，實際上課怎麼跑（老師怎麼看進度、怎麼派小老師）需要補。

---

## 6. 這份 spec 值得肯定的地方（別因為偏離就否定 spec）

這份 spec 的品質其實**高於多數**，以下都應保留並回灌到成品：

- **M1–M5 動作技能週原則**（動作品質、失誤靜音、進度視覺化、判定放大、反饋即節奏）——對 6–7 歲極到位。
- **Hook / Repeat / Deliver** 三段式承諾-心跳-兌現結構。
- **反硬 A 對照表**（每種作弊手法配一個擋法）。
- **明確資料結構**與**驗收標準對照表**（每站完成條件＋動作品質檢查）。
- **判定放大、CURSOR-PULSE 游標呼叫**等對小一操作特性的細膩照顧。

## 7. 如果我來寫 spec，我會補強的 12 點（核心建議）

> 前提：上面第 6 節的優點全部保留。以下是為了**避免這種「無人察覺的大偏離」**再發生。

1. **開頭放「正典聲明＋追溯表」**：本 spec 綁定哪個實作檔？任何偏離必須回填 spec（change control）。一句「spec 是唯一真相，程式跟著 spec 走；要改方向先改 spec」。
2. **可數名詞/單位表**：站、關、回合、次、圈——全部標準化。避免像本專案「10 關」被解讀成「10 回合」或「10 站」的歧義。
3. **明列 In / Out of Scope**：「本週只教 click/move/trace；雙擊/右鍵/滾輪/拖拉屬 W0X，**不在本週**」。範圍邊界寫死，範圍蔓延就有依據擋。
4. **跨週技能地圖**：W01–W06 每週教哪些滑鼠技能，一張總表。讓單週 spec 不會各自把別週的東西吸過來。
5. **完成的定義（DoD）＋參考範例**：把「像專業遊戲／情緒價值給滿」翻成**可勾選條目**＋一張參考畫面/影片。定性形容詞一定要配可驗收定義。
6. **「防挫折」與「反硬 A」拆成兩張獨立表**（成品把兩者混在一起）：對 6–7 歲，先定義「失敗絕不懲罰」的**全域不變式**，再定義防假動作。
7. **互動詞彙表當單一真相**：一次定義「右鍵＝橘色＝『點右鍵』」「滾輪往前＝上升」「雙擊＝×2＝藍色」。本專案出現過「左右鍵文案不對稱」「滾輪方向不一致」的 bug，就是因為沒有這張表。
8. **認知/識字負荷預算**：小一 spec 標明每畫面最多幾個字、是否需要識字；若引入閱讀理解，標注前置識字能力假設。
9. **時長預算表**：每站預估分鐘 × 反覆次數 = 加總，對齊課堂 90 分，避免超時。
10. **老師端＋翻轉教育列為一級需求，不是附錄**：URL/切換方式、即時同步、是否包含在交付範圍，寫清楚。
11. **素材/AI 生圖規格**：風格指南＋透明背景＋自動裁切＋**未載入要能退回向量**（成品這塊做得好，應把它變成硬需求寫進 spec）。
12. **跨切面需求第一版就寫**：存檔續玩／帳號綁定／金幣入錢包／資安。本專案「存檔續玩」是後期才補，導致 schema 反覆改與版本遷移（VERSION 4→5→6）。

## 8. 建議的下一步（一個決策，兩條路）

先花 30 分開一次對齊會議，決定 **P1U-W01 的正典是哪一套**：

- **若採「海底版」為正典** → 更新 spec 對齊現況；把 spec 的 M1–M5／老師端／翻轉教育／時長估算補進成品；把多出來的滑鼠技能**拆回 W02–W06**，讓 W01 回到合理的首週份量。
- **若採「登入版」為正典** → 現有成品其實更接近「模組一總複習/期末統整」，可改標為 P1U-W0X 或期末關卡；另外再依 spec 做出真正的 W01 登入版。

無論選哪條，**產出物都一樣**：spec 與實作檔互相指向、有版本、有追溯表——之後任何一方變動，另一方都跟著更新。
