# P1U 低年級教育遊戲設計準則（W02 spec vs 成品 回饋）

> 給課程 / spec 團隊。本文由「P1U-W02 王城故事線 分鏡 spec（2026-07-02）」與最終上線成品的逐項比對萃取而成，適用於未來所有 P1U / 低年級（注音世代）教育遊戲的 spec 撰寫。
> 產出方式：六個設計視角（敘事 / 難度 / 上手 / 獎勵 / 美術 / 工程）獨立比對 → 統整 → 完整性審查校正。

---

## 一句話總結

**spec 的「內容與敘事」幾乎全被忠實實作、事後證明是對的地基；但凡是 spec 去規定「底層輸入、失敗代價、難度數值、狀態與資料」的地方幾乎全被實測推翻**——因為那些是「從理論推得、未經目標年齡實玩、也未考慮真實課堂裝置 / 網路 / 帳號」的假設。

教訓很集中：**spec 要把「內容 / 敘事 / 視覺」寫死寫具體，把「難度 / 失敗 / 手感」標成「假設，待實玩校準」，把「狀態 / 資料 / 容錯」升級成一等公民正式章節。**

## 「spec 原意 vs 成品」關鍵偏離對照表

| spec 原意 | 成品最後 | 為什麼改 | 給 spec 的啟示 |
|---|---|---|---|
| 戰鬥控制＝左鍵 / 右鍵 / WASD（spec 自訂對照表） | 對齊 W01 全滑鼠：攻擊=右鍵、重擊=左鍵×2、防禦=左鍵、閃避=左鍵拖曳 | spec 自陳控制是「推得」、W01 未入文件；同系列第二週不該讓孩子重學相反手勢 | 底層輸入標「推定值」，寫死「與已上線成品衝突一律以成品為準」，附正典實作檔路徑 |

## 核心設計準則

### 一、敘事與教學理解
1. **每段劇情要交代「因」不只「果」。** 場景 spec 除了「發生什麼（果）」，必須規定「為什麼（因）」透過哪個互動被揭露；不得只呈現結果就進理解題，否則理解題退化成背圖。
2. **把「問對問題」設計成教學動作。** 需揭露背景的 NPC 對話寫成「多輪、每輪一個正解問題＋似是而非干擾」：問對才推進並揭露因果，問錯給非致命、不推進的軟回饋。天然對齊 AI 素養課「會問問題」。
3. **理解檢核分兩層：隱性建構＋顯性複述。** 先在互動中隱性檢核（選對問題＝證明懂處境），再用泡泡題顯性複述（剛剛整段發生什麼）；顯性題資訊必須先在隱性層被孩子主動接觸過，避免「用考試教學」。
4. **防硬 A 是理解檢核的效度前提，鎖時＋洗位置缺一不可。** 任何「點對進下一關」的題：點錯鎖 2+N 秒（連錯升冪）、每次點錯重洗選項位置、非致命不扣血、正解恆藏於 ≥3 個似是而非干擾中。只鎖時擋不住記位置，只洗位置擋不住高速輪點。
5. **干擾項＝同場景合理但與剛演出的因果矛盾。** 禁止洩題式或明顯荒謬選項，確保量到的是理解而非常識排除。
6. **敘事深度與「文字極簡」會衝突，取捨要在 spec 內明寫。** 鋪因果就得寫完整句、突破「≤5 字」；全注音只解決「讀得出音」不解決「讀得懂意」。spec 要明標「識字 / 閱讀負荷預算」與緩解手段（全注音 / 關鍵句配圖 / 語音），由 spec 端做取捨，不留給實作端即興。

### 二、難度與戰鬥手感
7. **命中要有重量：零代價的成功等於沒手感。** 「失敗代價」設成可調旋鈕（0→輕→重），預設值須經實玩，而非預設歸零。心流需要「輸的風險」作張力；保護靠安全網層級，不是抽掉所有代價。
8. **安全網與懲罰分兩層。** A 層終局保護（0HP 不淘汰、不退回更前進度、永遠能重試同關）**寫死不可調**；B 層即時懲罰（單拍扣血 / 斷連段 / 本關重來）逐關可調。難度只動 B，審查只查「B 有沒有動到 A」。
9. **難度成長要有「量」也要有「質」。** 量＝逐關數列（連段串長 1→3、縮圈速度、對手攻擊頻率）；質＝每關引入一個新機制（反向躲 / 全屏閃 / 破甲 / 二階段暴怒）。只調量會無聊，換新招才有學新東西的成長感——兩條線都要寫進 spec。
10. **留技能天花板：難度可被練習與精通，而非二元通過。** 主操作定義「新手能過 / 高手能炫」兩條標準（如連段＋PERFECT / combo），重複遊玩＝練習——難度曲線就是教學曲線。
11. **「重來單位」是一級難度參數。** 顯式定義失敗後要重玩多長（單拍 / 本小關 / 整場 / 整關）；長 boss 戰要有檢查點或階段保留，避免「一失誤重打兩分鐘」逼近耐挫上限。（本案 0HP→整隻章魚重打，對 14HP 的章魚王已接近上限，是需要標記的風險。）
12. **保底出口是低年級不變式，拿掉要補等效出口。** 難度可提高但不可移除「非懲罰性前進路徑」。若決定移除跳關鈕，須以 change-control 記錄決策 / 風險，並補「自適應出口」（失敗 N 次自動放寬該拍 / 慢動作預告 / 老師端一鍵放寬）而非留死路。
13. **難度承諾必須跨裝置等價，或明文寫出補償。** 為每個目標輸入裝置（滑鼠 / 觸控 / 螢幕尺寸）定判定寬容度與失敗補償。（實測程式碼發現：觸控裝置上魔法 / 閃避 miss 被自動改判命中——這種隱藏補償 spec 完全沒提，導致「極度溫和」的承諾在滑鼠上其實不成立、觸控上又過鬆。補償要寫進 spec，別藏在程式碼裡。）

### 三、上手引導與低年級適配
14. **新輸入方式必有專屬教學關，一次只練一個。** 用機制強制「當下唯一正確操作」（如做完動作自動回中立位），把「學會操作」與「理解劇情」兩種認知負荷分開。
15. **跨週連載每集開場必做「上集回顧＋本集目標預覽」。** 一句一頁回顧上集、點出本集衝突、可視化本集目標（秀 N 個空欄位）。孩子隔週不記得劇情，沒回顧＝每週重啟、主線失黏。
16. **判定區與視覺尺寸分離，判定恆 ≥ 視覺 2–3 倍。** 每個可互動元素標兩個尺寸；點擊判定半徑建議 ≥100px。低年級游標與方向控制會飄，「明明到了卻沒反應」對上手是致命的。
17. **上手第一秒不能白屏或卡死；且要先寫清「阻塞 vs fail-open」的優先序。** 進場前有載入畫面＋進度可視化。本案同時有「圖片全載完才能開始」與「載入失敗 / 逾時放行」兩條——spec 必須明訂優先序：**預設阻塞等待，但設逾時（如 10 秒）強制放行，且缺圖退回向量後備**，三者一起寫，別讓實作端猜。
18. **控制方案跨週統一，以「前一週已上線成品」為單一真相。** spec 憑空自訂的新對照表不得覆蓋既有肌肉記憶；同一動作全程綁同一指令，強化版用「同鍵加強」而非換鍵。

### 四、獎勵結構與學習動機
19. **獎勵全程可視、即時、且顯示總量。** 常駐 HUD 顯示「目前/總量」，每次獲得給即時動畫（跳動數字＋「+N」浮字＋音效）；不可只寫在結局。「還差多少才滿」是撐長流程動機的主引擎。
20. **收集閉環要做成頭—中—尾一條線。** 開場「預告」目標（秀 N 個空欄位）→ 過程逐關「點亮」進度（每個收集物彼此可辨識：顏色 / 名字，各對應一關）→ 結局「親手交付」儀式關（逐一放定位＋逐一回饋＋集滿慶祝）。三段是同一條主線，spec 要當一條寫、不要拆散。
21. **獎勵發放點對齊真正的學習成就。** 每份獎勵綁一個具體學習動作（答對理解題 / 打敗某關），禁止純點擊、純時間、純進場發獎。
22. **把「教別人」寫成有獎勵的可執行步驟。** 翻轉教育 TA 不只當結局標語：落地成明確按鈕＋利他任務（去教一位同學）＋實質獎勵背書＋獨立交易 ID。
23. **跨集累積經濟優於單集一次性榮譽；頒獎是必須被設計的情緒高點。** 優先選可跨週累積、綁學生帳號、老師端可用的通用貨幣（單集徽章作點綴）。頒獎用獨立儀式化畫面：英雄稱號＋多感官慶祝＋入袋播報＋集滿總覽，不可一句文字草草結束。

### 五、美術一致性與資訊視覺化
24. **一角一圖、全程複用（角色即識別碼）。** 每個角色 / 地點只產一張權威去背圖，在所有場景複用；禁止同一實體另畫第二版。跨頁一致是「剛剛那隻＝現在這隻」與理解題成立的地基。
25. **難度要有一條可量化的視覺主軸。** 「越強越難」綁定單調遞增的視覺變數（體積優先，其次深色 / 特效密度 / 佔畫面比例），在 spec 寫成數列（如 sizeK 0.78→1.34）。難度的可讀性本身就是難度設計的一部分——孩子讀不懂數字，但看得懂「這隻更大更黑更兇」。
26. **關卡 ≥4 就聚合成一張可俯瞰的地圖。** 用每關代表圖當 marker、路徑串起，回答「我在哪、去過哪、還要去哪」，把抽象「第幾關」翻成空間敘事。
27. **marker 用 done/active/future 三態編碼，且 future 保持可辨識。** 已完成（綠+勾）、進行中（高彩度脈動+光暈+標籤，唯一焦點）、未解鎖（壓暗但不全黑）。全黑會抹掉預告價值，全亮會失去焦點導引。
28. **canvas 承載資訊的圖示一律自繪，禁賭系統 emoji。** emoji 在部分裝置渲染成豆腐字＝資訊遺失；貨幣 / 狀態徽章 / 記號 / 箭頭用向量或去背 PNG。
29. **一致的美術要配一致的降級路徑。** 每個 sprite 有向量 fallback，去背執行期自動裁切，載入失敗與逾時放行。美術一致性若無容錯，缺一張圖就開天窗，比沒美術更糟。

### 六、資料完整性與工程可靠度
> 註：本節多為「任何有雲端存檔＋計分的多裝置 web app」都需要的通用工程可靠度要求，不是低年級教育遊戲特有。但它們是 spec 過去整段留白、事後被迫補的東西，所以仍列為 spec 必寫章節。

30. **分數是有金流語意的資料，發獎一律要求冪等（與「能不能重玩」無關）。** 每次入帳有「學生×關卡×事件」決定的固定交易 ID，發放前查重；明文寫出「重整 / 斷線重連 / 回關重玩 / 多裝置 / 整章重玩」時都不得重複入帳或漏計。**這條保證的是「同一成就只計分一次」，它本身不禁止玩家重玩。**
31. **「計分冪等」與「玩家重玩權」是兩個決策，必須拆開。** 為了分數正確而「同一關不重複入帳」＝對、且與保底出口不衝突。但「玩家能不能重玩已完成關卡（練習用）」是另一個產品決策——本案選擇「完成即鎖、不可重做」，這**與準則 12「永遠有非懲罰性出口」在立場上對撞**。spec 團隊要明確二選一並記錄理由：(A) 分數冪等＋允許無計分的自由重玩（推薦，兼顧練習與保底）；(B) 分數冪等＋完成即鎖（防重刷、但犧牲重練與保底出口）。不要像本案一樣兩條準則各寫一半、互相矛盾。
32. **「進度」定義成單調不可逆的分數，衝突取最遠。** 把 phase 偏序＋完成數壓成單一可比數值：本地 vs 雲端衝突取進度較遠者、絕不盲蓋。跨裝置還原、防倒退共用同一裁決依據。
33. **把失敗 / 離線 / 缺資產當正常狀態寫進 spec。** (1) 網路 / 帳號不可用時本地為即時真相、雲端為背景備份、未就緒靜默略過；(2) 資產載入失敗有向量 / 文字後備並照常進行；(3) 載入有逾時上限強制放行。每條回答「當 X 掛掉時學生看到什麼、能不能繼續玩」。
34. **「全注音 / 全在地化排版」是子系統，不是排版開關。** 附「渲染驗收清單」：逐字注音對位、聲調位置、非中文字（英數 / 標點 / 空白）間距、過長自動縮排、emoji 是否可用。對注音世代這是「看不看得懂」的硬需求。可複用的注音渲染器應登記為共用元件，未來 spec 直接引用。
35. **「成品即真相」寫成正式衝突裁決條款。** 從他章 / 前作「推得」而非親測的段落（尤其底層輸入 / 失敗處理 / 難度數值）標為「推定值」，並寫明「與成品不符一律以成品為準」。清掉殘留過時註解（如本案 header 一度仍寫「全程可跳關」，功能已移除）。

## spec 團隊以後該怎麼寫 spec：Checklist

**A. 條款分級（每條都要標）**
- [ ] 標「規定（親測 / 內容敘事，工程必須照做）」或「推定（從他處推得，成品優先）」
- [ ] 推定條款寫明「與成品衝突以成品為準」，交付時附「需向成品 / 前作核對的推定清單」
- [ ] 未經目標年齡實玩的難度 / 手感決策，標「假設，待實測校準」，並排一輪低年級實玩校準節點

**B. 敘事與理解**
- [ ] 每個場景設「因果欄」：分寫「玩家看到的果」與「必須被揭露的因」＋指定揭露互動
- [ ] NPC 對話寫成結構化表格：每輪列「正解問題→揭露背景」＋2~3 個「干擾問題→軟回饋」，標該輪負責的主線因果節點
- [ ] 理解檢核明訂兩層（隱性問對問題 / 顯性泡泡題），附「顯性題資訊已在隱性層出現」一致性檢查
- [ ] 每道理解題寫死防硬 A 參數：鎖時 2+N、連錯升冪、點錯必重排、干擾 ≥2、非致命不扣血
- [ ] 附干擾項規則＋反例庫（同場景合理但與因果矛盾；禁洩題 / 禁荒謬）
- [ ] 編「識字 / 認知負荷預算」：每畫面最長句、是否全注音、前置識字假設

**C. 難度（用表格，不用哲學句）**
- [ ] 動作集當**封閉集合**明列（就這幾個動作 / 這幾種輸入）；新增動作或輸入視為破壞「單一真相」的變更，走 change-control
- [ ] 每個動作關填表：【失敗處置：換拍 / 扣血 / 斷連段】【重來單位：單拍 / 本關 / 整場】【扣血量】【0HP 後果】
- [ ] 難度章節採「安全網分層」：先寫死 A 層（終局保護恆定），再開放 B 層（即時懲罰逐關可調）
- [ ] 難度成長寫**兩條線**：量的數列（連段 / 速度 / 頻率）＋質的機制（每關新招清單）
- [ ] 每個主操作定義「新手線 / 精通線」
- [ ] 一張「輸入裝置 × 判定寬容度」矩陣（滑鼠 / 觸控 / 螢幕尺寸各一列，含命中半徑 / 時間窗 / 失敗補償）
- [ ] 保底出口與難度分兩張表；移除任一保底走 change-control 並補「自適應出口」

**D. 上手與適配**
- [ ] 「跨週控制真相表」：每動作↔輸入↔顏色↔圖示，衝突規則「以前一週成品為準」＋正典實作檔路徑
- [ ] 引入新輸入 → 硬性要求獨立教學關（機制上當下唯一正確操作）
- [ ] 連載「開場序章」模板：一頁上集回顧＋一頁本集衝突＋一頁可視化目標，全注音、一句一頁
- [ ] 每個可互動元素標「視覺尺寸 / 判定尺寸」，判定 ≥ 視覺 2–3 倍、點擊 ≥100px
- [ ] 載入與素材保底列為可勾 DoD，並寫明「阻塞等待→逾時放行→向量後備」優先序

**E. 獎勵**
- [ ] 「獎勵結構表」：發放點 / 對應學習成就 / 種類數量 / 是否可視化 / 交易 ID，數字加總得出明確總量（HUD 分母）
- [ ] 「獎勵 HUD」列必備元件＋行為（常駐位置 / 目前總量格式 / 跳動動畫 / +N 浮字 / 過場顯示）
- [ ] 收集型主線寫成頭中尾一條：收集物清單（數量 / 顏色 / 名稱 / 對應關）＋常駐進度＋結局親手歸還儀式關
- [ ] TA（小老師）升格為必填收尾任務：按鈕文案 / 利他任務 / 額外獎勵 / 獨立交易 ID
- [ ] 明選獎勵載體（跨集金幣為主 / 單集徽章為點綴），標註綁定 `coins/{uid}`、老師端可見
- [ ] 「頒獎畫面規格」：英雄稱號 / 慶祝特效 / 入袋播報 / 集滿總覽，獨立畫面

**F. 資料與工程（新增正式章節）**
- [ ] 「資料與狀態」章節：分數 / 進度 / 完成狀態存哪、綁誰（uid 來源）、跨裝置如何還原、衝突如何裁決
- [ ] 每種獎勵列「發放表」：事件→固定交易 ID→金額→是否可重複，明訂「同一成就只計分一次」
- [ ] **先回答「分數是不是學習憑證」**，再決定「玩家能否重玩已完成關卡」；把「計分冪等」與「重玩權」拆成兩個決策各自寫清楚（別互相矛盾）
- [ ] 「進度量尺」：phase 偏序＋完成數壓成單一可比數值，衝突取最遠
- [ ] 「失敗與離線行為」獨立於難度章節：操作失敗看到什麼 / 0HP 出口 / 斷線能否續玩 / 缺檔後備 / 載入逾時
- [ ] 每個非標準在地化需求（注音 / 大字 / 語音）附「渲染驗收清單」

**G. 美術與資訊視覺化**
- [ ] 角色圖鑑加「複用清單」欄：每角色 / 地點權威去背圖出現在哪些畫面，一實體只產一張
- [ ] 「越強看起來越強」改成數字：每關視覺尺寸係數數列＋逐關疊加特效清單
- [ ] 關卡 ≥4 強制含「總覽地圖分鏡」（marker 佈局 / 路徑走向 / 每 marker 用哪張圖 / 戰鬥點記號）
- [ ] marker done/active/future 三態視覺規格表，明文要求 future「壓暗但仍可辨識」
- [ ] 美術資產標實作方式（向量 / 去背 PNG）＋硬規則「canvas 承載資訊圖示禁用 emoji」＋「每 sprite 向量 fallback」
