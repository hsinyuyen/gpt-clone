## 0. 一句話結論

> **這次不是「規格沒被執行」，而是「規格被完整執行了，然後被真實學生推翻」。**
> spec 寫的「持續同步兩手戰鬥、難度地板高於 W02」被做成 v1 彈幕版，實測**兩個難度學生都過不了**，整個打法重做成 v2 節奏拍 QTE。
> 核心問題出在一條 spec 條文：**難度目標是用「相對於上一堂」定義的，而不是用「目標學生的通過率」定義的。**

W01 的問題是「規格與成品是兩套設計」；W03 的問題更隱蔽 —— **規格內部自洽、開發忠實執行、驗收清單全過，但產品仍然失敗**。這份文件要回答的是：spec 該怎麼寫，才能在紙上就攔下這件事。

## 2. 五個關鍵偏離與原因

### 2-1 難度：spec 用「相對基準」訂目標，是這次失敗的根因 ⚠️ 最重要

spec 寫得非常明確：

> 難度定位【規定】：⋯⋯若做出來感覺跟 W02 第 3 區差不多 → **未達標，退回加難**。

這條規則有兩個問題：

1. **基準是上一堂課，不是學生。** 只要每堂都「比上一堂難」，難度就會單調遞增，直到超出學生能力 —— 而且**每一堂單看都合規**。
2. **它把「太難」變成了合規行為。** 開發者做完自測覺得「好像沒有很難」，依照這條規定必須**退回加難**。v1 彈幕版就是這樣被推上去的。

二修的修改理由更能說明問題：

> 二修⋯⋯原因：難度上修 —— 前一版誤把前哨寫成「波 1 只攻、波 2 只躲」的漸進入門，難度反而低於 W02 第 3 區。

**被二修否定掉的那一版（漸進入門），才是最後真正能用的設計。** 一修的直覺是對的，是被「必須比上一堂難」這條規則推翻的。

> **建議**：難度條文一律改用**可觀測的學生行為**當基準，例如「目標班級 80% 學生在 15 分鐘內能打完第 1 關且不需要老師介入」。相對難度只能當**參考**，不能當**驗收條件**。

### 2-2 「持續同步」對小三小四是認知超載

spec 的核心賣點是「左手不停走位、右手不停戰鬥，兩件事同時且持續」。實作出來後的真實情況是：學生必須同時追蹤子彈軌跡、自己的位置、弱點出現位置、弱點要用哪個滑鼠鍵 —— **四條資訊流並行**。

v2 的解法是把「同時」從**持續**壓縮成**瞬間**：
- 每一拍只有一件事，但那一件事**必須兩手一起**（按住 `D` ＋ 點左鍵）
- 學習目標（兩手協同）完全保留，認知負荷大幅下降

> **建議**：spec 提到「同時進行 A 與 B」時，要註明是**持續同時**還是**瞬間同時**，兩者難度差距極大。低年級預設用瞬間同時。

### 2-3 A 層安全網把「跳波鈕」寫死，方向錯了

spec：

> A 層（保底・寫死不可調）：⋯⋯**全程有跳波鈕**。

但實際教學需求是**完全相反**的 —— 使用者明確要求拿掉所有跳過按鈕，因為學生會一路點過去、不看劇情也不做檢核。

A 層的原始用意是「**保護學生不被卡死**」（0 血不淘汰、可重試、連敗自動放寬），這些都對。但「跳波鈕」不屬於這一類 —— 它是「**允許略過學習內容**」，那是老師的教學決定，不該寫進不可調的 A 層。

> **建議**：A 層只保留「不會卡死、不會倒退、不會羞辱」三類條款。「能否跳過」移到 B 層或老師端開關，預設**關閉**。

### 2-4 §七 寫了「綁 uid」，但沒有人檢查它有沒有真的做到 → 正式資料被污染

spec §七【規定】：

> 資料與狀態：進度⋯⋯存 localStorage + 帳號雲端；**綁 uid**；跨裝置取進度較遠者。

實作的存檔鍵是固定字串（`game:p1uw03`），**沒有綁 uid**。後果：

1. A 同學在教室電腦玩完 → 存檔留在瀏覽器
2. B 同學在同一台電腦登入、開同一個遊戲 → **讀到 A 的進度、顯示已完成**
3. 接著同步到雲端 → **B 的 Firestore 紀錄被 A 的進度覆蓋**，換電腦也錯，老師後台看到的也錯

實際盤查（交叉比對「進度與他人位元級相同」且「該遊戲零金幣紀錄」）找到 **7 筆污染紀錄**，跨 3 個遊戲、涵蓋 W01/S2W01/S2W02 —— **這不是 W03 的 bug，是全系列共用的 bug，W03 只是讓它被發現**。

值得注意的是：**同一份 spec 的§七也寫了「金幣冪等」，而那條有被實作**。差別在於金幣冪等在遊戲內看得到效果（重複破關不會加錢），uid 綁定則要**兩個學生共用一台電腦**才會顯現 —— 開發與自測都碰不到。

> **建議**：凡是「單機自測碰不到、要多使用者/多裝置才會顯現」的條款，spec 要在 DoD 標成**必須用指定情境驗收**，並寫出情境腳本。例如：
> ```
> □ 共用電腦驗收：A 帳號玩到第 2 關 → 登出 → B 帳號登入開同一遊戲
>   → B 必須是全新進度，且 B 的雲端紀錄不得出現 A 的資料
> ```

### 2-5 素材限制（一圖複用）反而限制了體驗

spec：

> 素材【規定】：先鋒**全為小章魚兵（一圖複用）**＋1 隻前哨隊長（略大）⋯⋯靠「數量與招式」而非體型製造難度。

這條的動機（控制美術成本）在 AI 生圖的前提下已經不成立。實際交付的是**每關不同章魚**＋8 張整幅劇情圖，生成成本約幾分鐘，而「每關換一隻新的」對小朋友的推進感遠大於「同一隻變多隻」。

> **建議**：素材條款改寫成**下限**而非**上限**（「至少 N 種、可視成本增加」），並註明 AI 生圖的實際成本量級，避免用舊的成本假設限制設計。

## 3. spec 沒想到、但實作長出來且值得回填的三個機制

| 機制 | 內容 | 為什麼值得寫進 spec |
|---|---|---|
| **逐段理解檢核** | 5 頁開場劇情＋3 段幕間，**每一段後面接一題泡泡選擇題**，答對才能繼續；答錯鎖 2+N 秒並重新洗牌位置（防硬 A） | spec 的 Hook/Repeat/Deliver 只保證「有講故事」，不保證「有聽進去」。加上檢核後，學生無法一路點過去 |
| **關間 WASD 引導段** | 關卡之間有一段獨立的海溝，用 WASD 游到下一隻章魚那裡才開打，沿路有金幣 | 把「走位」從戰鬥中拆出來單獨練，是 spec「開場即兩手」做不到時的正確替代方案 |
| **關主大招連段 QTE** | 王血過門檻時全螢幕大招，要照順序按完 `W→D→S→A→右鍵→左鍵→滾輪`；按錯從頭、逾時扣 3 血 | 這才是 spec 想要的「招式更雜、節奏壓力」的正確載體 —— 壓力集中在**短時間高潮**，而不是**全程持續** |

## 4. 建議回填 spec 的條文（可直接取用）

```diff
- 難度定位【規定】：⋯⋯若做出來感覺跟 W02 第 3 區差不多 → 未達標，退回加難。
+ 難度定位【規定・以學生為基準】：
+   目標班級（小三/小四）在無老師介入下，80% 能於 15 分鐘內完成第 1 關；
+   全堂 90% 能完成至少 3 關。相對於前一堂的難度僅作參考，不作為驗收條件。
+   ⚠️ 未達上述通過率 → 一律往下調，不得以「比上一堂簡單」為由拒絕。

- 兩手分工核心：左手 WASD 不停走位，右手滑鼠不停戰鬥 —— 兩件事同時、持續發生。
+ 兩手分工核心【規定・瞬間同時】：
+   每一次判定都要求「左手按住指定鍵 ＋ 右手做對滑鼠動作」同時成立；
+   但一次只判定一件事，不要求持續雙軌操作（低年級認知負荷上限）。

- A 層（保底・寫死不可調）：⋯⋯全程有跳波鈕。
+ A 層（保底・寫死不可調）：0 血不淘汰、不退更前進度、永遠能重試、連敗 N 次自動放寬。
+ 跳過機制【規定】：預設**不提供**跳過（劇情/教學/關卡皆是）；
+   若課堂需要，僅能由老師端開啟，不做成學生端常駐按鈕。

+ 理解檢核【規定・新增】：每一段劇情後接一題本地批改的理解題，答對才能繼續；
+   答錯需鎖定數秒並打散選項位置（防硬 A）。劇情不得是純點擊過場。

+ 多使用者驗收【規定・新增】：共用電腦情境必須實測 —
+   A 帳號留下進度 → B 帳號登入 → B 必須全新開始，且 B 的雲端紀錄不得含 A 的資料。
+   所有本地存檔鍵一律綁 uid；登出時清除本機遊戲存檔。

- 素材【規定】：先鋒全為小章魚兵（一圖複用）＋1 隻前哨隊長。
+ 素材【規定・下限】：每一波至少一種可辨識的敵人外觀（AI 生圖成本已可負擔）；
+   劇情每一頁配一張整幅場景圖。上限依實際成本自行增加。
```

## 6. 給規格團隊的四條流程建議

1. **難度用學生通過率定義，不用相對於上一堂定義。** 這次的失敗完全來自這一條。相對難度只能寫在「設計意圖」，不能寫在「驗收條件」。
2. **區分「持續同時」與「瞬間同時」。** 這兩個詞在 spec 裡長得很像，難度差好幾倍。低年級一律預設瞬間同時。
3. **橫切需求（存檔、帳號、金幣、離線）要附驗收情境腳本。** 「綁 uid」四個字有寫，但沒人能在單機自測中發現它沒做到 —— 結果污染了正式資料庫。這類條款要寫成「怎麼測」而不是「要做到」。
4. **spec 修訂時，要記錄「被否決的方案」與否決理由。** 這次二修否決掉的漸進式設計，最後才是對的。如果 spec 保留了否決紀錄，回頭檢討時能更快找到問題（而不是重新發明一次）。
